# 技术选型文档

## 1. 文档目标

本文档用于明确语雀类在线文档产品当前阶段的整体技术选型，统一前后端开发基线，为后续项目初始化、数据库设计、接口设计与部署方案提供依据。

## 2. 选型结论

当前技术方案确定如下：

- 前端：React + Vite + TypeScript + Ant Design
- 路由：React Router
- 前端状态管理：Zustand
- 服务端状态管理：TanStack Query
- 前端样式方案：CSS Modules
- 编辑器：后续在 Tiptap 或 Slate 中最终确定
- 后端：.NET 10
- 数据库：PostgreSQL 18

## 3. 前端技术选型说明

### 3.1 React

选择 React 作为前端框架，原因如下：

- 生态成熟，适合复杂交互型 Web 应用。
- 适合构建组件化页面，利于文档产品中大量复杂业务模块复用。
- 与编辑器生态兼容度高，后续接入 Tiptap 或 Slate 风险较低。

### 3.2 Vite

选择 Vite 作为前端构建工具，原因如下：

- 本地开发启动快，适合前期高频迭代。
- 构建配置相对轻量，适合中型前端工程起步。
- 与 React、TypeScript 组合成熟稳定。

### 3.3 TypeScript

选择 TypeScript 作为前端开发语言，原因如下：

- 便于维护复杂业务模型，如权限、文档结构、评论、版本等。
- 适合多人协作，降低接口字段和状态结构误用风险。
- 为后续抽象业务组件、Hooks、接口层提供更好约束。

### 3.4 Ant Design

选择 Ant Design 作为基础 UI 组件库，原因如下：

- 组件完备，适合中后台与复杂业务界面快速落地。
- 中文生态成熟，便于快速查阅资料和解决问题。
- 对表单、表格、弹窗、抽屉、通知等场景支持较好。

使用策略如下：

- 管理页、设置页、权限页、成员页优先使用 Ant Design。
- 文档阅读页、编辑器区域、知识库导航等核心内容区域以自定义样式和业务组件为主。
- 所有高频业务组件需二次封装，避免未来替换成本过高。

### 3.5 React Router

用于管理路由体系，承担公开页、业务页、后台页的路由组织，并与权限控制结合。

### 3.6 Zustand

用于管理轻量全局状态，如当前用户、当前团队、布局状态、偏好设置等。其优势在于简单、心智负担低、适合中型前端项目。

### 3.7 TanStack Query

用于管理接口查询与变更状态，负责缓存、失效、重试、并发请求与异步状态流转。适合文档详情、评论列表、知识库列表、搜索结果等服务端状态管理场景。

### 3.8 CSS Modules

用于实现样式隔离，避免大型前端项目中全局样式污染问题。适合与组件分层方案结合，建立稳定的局部样式体系。

### 3.9 编辑器待定

编辑器暂不立即锁定，后续在 Tiptap 与 Slate 之间选择，原则如下：

- 支持较强扩展能力。
- 支持文档内容结构化表达。
- 兼容评论、历史版本、导入导出等后续能力。
- 能适应未来实时协作或复杂内容块扩展需求。

## 4. 后端技术选型说明

### 4.1 .NET 10

后端选择 .NET 10，原因如下：

- 适合承载权限、版本、审计、评论、文档管理等复杂业务逻辑。
- 强类型与工程化能力较适合中长期演进。
- 在事务处理、模块化单体架构、后台任务等方面较稳。
- 与 PostgreSQL 配合成熟。

建议首版采用模块化单体架构，而不是一开始拆微服务。推荐先按业务域划分模块：

- 认证与用户
- 团队与组织
- 知识库
- 文档
- 评论
- 权限
- 通知
- 搜索
- 审计日志

## 5. 数据库选型说明

### 5.1 PostgreSQL 18

数据库选择 PostgreSQL 18，原因如下：

- 非常适合用户、团队、知识库、文档、权限、评论、版本等关系型业务数据。
- 支持 JSONB，可用于承载部分扩展字段、编辑器结构化内容、审计附加信息。
- 索引能力、事务能力、稳定性适合长期演进。
- 可在前期承担基础全文检索能力，后期再根据压力扩展专门搜索引擎。

建议使用 PostgreSQL 管理以下核心实体：

- users
- teams
- team_members
- repositories
- repository_members
- documents
- document_versions
- comments
- attachments
- notifications
- operation_logs

## 6. 推荐配套基础设施

虽然当前核心选型已确定，但建议在架构上预留以下配套能力：

- Redis：用于缓存、会话辅助、限流、异步任务协同。
- 对象存储：用于图片与附件存储。
- 搜索扩展位：首版可先用 PostgreSQL 基础检索，后期扩展 Elasticsearch 或 OpenSearch。
- 实时协作扩展位：后续可基于 WebSocket 或 SignalR 增强协同编辑能力。

## 7. 技术组合优势

该组合对当前项目的优势主要体现在：

- 上手快，便于尽快交付首版。
- 技术风险低，前后端都有成熟生态支撑。
- 能覆盖文档产品的核心复杂度，如权限、目录树、评论、版本、搜索、阅读与编辑分离。
- 未来可平滑扩展到实时协作、搜索引擎、对象存储、企业级权限体系。

## 8. 当前不建议的路线

### 8.1 前端不建议全自研 UI

首版不建议自己从零构建完整 UI 基础设施，否则会在表单、弹窗、表格、下拉、通知等基础组件上消耗过多时间。

### 8.2 后端不建议首版上微服务

当前阶段业务边界还在快速变化，首版采用微服务会提升沟通和部署复杂度，不利于快速迭代。

### 8.3 不建议过早绑定单一编辑器实现

编辑器是该产品的核心底座之一，但首版无需仓促定型，应在明确内容模型、评论方式、版本策略后再最终锁定。

## 9. 项目落地建议

### 9.1 第一阶段

- 初始化 React + Vite + TypeScript 前端项目。
- 接入 Ant Design、React Router、Zustand、TanStack Query、CSS Modules。
- 初始化 .NET 10 Web API 项目。
- 建立 PostgreSQL 18 数据库开发环境。

### 9.2 第二阶段

- 完成登录、工作台、知识库、文档详情的基础能力。
- 建立统一请求层、统一权限模型、统一错误处理机制。
- 完成核心数据库表结构设计。

### 9.3 第三阶段

- 接入编辑器。
- 实现版本、评论、搜索、分享等复杂能力。
- 逐步补充缓存、对象存储、后台任务与搜索增强能力。

## 10. 后端基座（Yuque_Backend）与裁剪约定

- **代码位置**：仓库内 `Yuque_Backend/`（.NET 10 + EF Core + Npgsql + Redis + RabbitMQ + SignalR 等）。
- **语雀业务扩展**：在现有分层上增加知识库 / 文档 / 版本等模块；**主键沿用雪花 `bigint`**；**文档正文列使用 `jsonb`**（见 `docs/database_schema.md`）。
- **库表约束**：语雀新建表 **不建数据库外键**，仅用业务 id 逻辑关联，与基座 `UseRemoveForeignKeys` 一致；**附件使用独立 `attachments` 表**（不复用 `File`），见 `docs/database_baseline_table_mapping.md`。
- **权限**：基座 **RBAC 仅用于管理端与系统设置**；**知识库与文档权限单独建模**（见 `docs/backend_baseline.md` §2）。
- **裁剪**：**已移除 AgileConfig 客户端启动逻辑**；连接串使用 **`ConnectionStrings:DefaultConnection`**（并兼容旧键 `PostgreSQL`）。**保留** `Yuque.MQService`、`Yuque.PlanTaskService` 以备文档同步与实时协作演进；**API 网关**（`Host/Yuque.Gateway`）不纳入当前解决方案与默认交付，单机开发以 `Yuque.WebAPI` 为主。
- **详细约定**：见 `docs/backend_baseline.md`。

## 11. 结论

本项目当前阶段的最优技术路线为：前后端分离，前端使用 React + Vite + TypeScript + Ant Design，配合 React Router、Zustand、TanStack Query 与 CSS Modules；后端使用 .NET 10；数据库使用 PostgreSQL 18。该方案兼顾首版交付效率、业务复杂度承载能力与后续扩展空间，适合作为语雀类在线文档产品的第一阶段正式技术基线。
